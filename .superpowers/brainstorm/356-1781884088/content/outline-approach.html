大纲实现 — 两种方案对比

技术实现层面的选择

A

AST 提取方案（推荐）

从已解析的 Document AST 中提取 Block::Heading 节点，使用 BlockWithSpan.span.start_line 获取行号，Heading.level 获取层级，Heading.inlines 渲染标题文本。

优点

  • 复用已有解析基础设施（RenderCache）
  • 行号精确（来自 parser 的 span 信息）
  • 不需要额外的文本扫描
  • 与编辑器内容完全一致

缺点

  • 依赖 AST 解析正确性
  • 语法错误时可能缺失部分标题
  • 空行前的标题可能边界不准
B

正则扫描方案

直接用正则 ^(#{1,6})\s+(.+)$ 扫描源码文本提取标题和行号。不依赖 AST 解析。

优点

  • 不依赖 AST 解析结果
  • 即使解析出错也能工作
  • 实现极简

缺点

  • 无法识别代码块内的 # 注释（误判）
  • 无法处理 Setext 风格标题（=== / ---）
  • 标题文本可能包含内联标记（**粗体**）未处理
  • 需要额外维护正则逻辑

推荐方案 A — AST 提取

使用已有的 Document.blocks 筛选 Block::Heading 节点：

/// 从 Document AST 提取大纲项
struct OutlineItem {
    level: u8,        // 1-6
    text: String,      // 纯文本标题（去掉内联标记）
    line: usize,       // 在源码中的行号
}

fn extract_outline(doc: &Document) -> Vec<OutlineItem> {
    doc.blocks.iter()
        .filter_map(|bws| match &bws.block {
            Block::Heading(h) => Some(OutlineItem {
                level: h.level,
                text: inlines_to_plain(&h.inlines),
                line: bws.span.start_line,
            }),
            _ => None,
        })
        .collect()
}

核心逻辑简单，复用已有基础设施。边缘情况（代码块内误判等）由 markdown parser 处理。
